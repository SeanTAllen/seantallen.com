GC Made Fast

This is a talk about Pony (more on Pony), and going fast, and garbage collection. Really this is a talk about message passing.

Most Engineers I Know

Most people don't think GC and fast. Most engineers I know think GC is slow and I can't blame them. Most of them experience GC as stop-the-world GC. And that gives you an experience like...
With a stop-the-world garbage collector you get nasty latency spikes that can destroy the stability of your system.

The

Message Passing

Bet

Message Passing
scales better than
Shared State

Shared state vs. message passing is an old argument (micro-kernel vs monolithic), and I'm firmly in the message-passing camp. But why? Not because message passing is inherently fast — it's because it enables something deeper.

Coordination
Kills
Performance

"Silence is Golden" / Helland's "Life Beyond Distributed Transactions": the fastest coordination is no coordination. The best way to go fast is to do as much as possible with local knowledge alone. Message passing serves this: actors communicate to share information, but they don't need to coordinate to *act* on that information. Gossip, not consensus.

Act Locally

Gossip Globally

This principle — act locally, gossip globally — turns out to apply all the way down to memory management.

The Hidden Cost of

Coordination

in Memory Management

The Coordination Tax

Atomic reference counting

Lock-free reclamation

Cache-line bouncing

In shared-memory languages: atomic reference counting is coordination — every copy/drop is a cache-line bouncing atomic op, forcing cores to agree on a count. Lock-free reclamation (hazard pointers, epoch-based) is coordination — threads must agree on when memory is safe to free.

Didn't Rust Solve This?

Single thread? Yes

Multiple threads? No

Rust gets this partly right: ownership and borrowing eliminate coordination for single-threaded access. But cross-thread sharing still requires runtime coordination — Arc, mutexes, channels. Rust solves aliasing at compile time within a thread but not across threads. (This sets up Pony's answer: what if you solved the cross-actor case at compile time too?)

Stop the World GC

Maximalist Coordination

In GC'd languages (Java, Go): stop-the-world is the ultimate coordination — every thread stops so the collector can get a consistent view of the heap. The conventional wisdom treats this as inevitable. More concurrency = more coordination over memory. Pick your flavor of coordination, but you can't escape it. But that's only true if your memory management requires global knowledge. What if it didn't?

Pony's Actors

No shared mutable state

No copying to share data

Quick Pony intro — only what's needed: actors as the unit of concurrency, no shared mutable state. The type system tracks not just ownership but *how* a reference can be used — mutable or immutable, shareable or unique. This is decided at compile time, not enforced at runtime, so there's no coordination cost.

Per-Actor Heaps

Per-actor heaps: each actor owns its own heap, collects independently. Collection is a local decision based on local knowledge. No actor needs to know what any other actor is doing. No stop-the-world. Ever. One actor collecting doesn't touch any other actor. Collection cost scales with the individual actor's heap, not the total program heap.

Message Passing
Is Cheap

Message passing is cheap: sending a message doesn't require atomic reference count updates or synchronization on the object — you just hand it over. The type system guarantees the sender gives up access, so there's no aliasing problem to solve at runtime. Immutable data can be shared across actors without copying — the type system guarantees no one can mutate it, so no coordination is needed.

Gossip

Cross-actor references — the one place where actors can't act purely locally: what happens when actor A sends an object to actor B and B holds onto it? Actors gossip reference information by piggybacking on application messages — no separate GC channel, no synchronization. Each actor uses the gossiped information to make its *own* local collection decisions. They share information but don't coordinate action. This is the "silence is golden" principle applied to GC: communicate when you must, but never wait for agreement to act.

Act Locally

All the Way Down

Coordination Is Slow, GC Isn't

"GC is slow" is really "GC that requires global coordination is slow." That's not an inherent property of garbage collection — it's a consequence of designing GC for shared heaps. This isn't to say it's without cost — everything you have a computer do has a cost. But you need to look at a system as a totality, not as individual pieces, to understand what is "fast." Per-actor GC has costs; they're just not coordination costs.

Go All In

When you commit fully to the message-passing model — actors with isolated heaps, no shared mutable state — GC stops requiring coordination. The kernel analogy revisited: just as microkernels gain isolation by committing to message passing, Pony gets GC properties that seem impossible if you're still thinking in shared-heap terms. The commitment to the model is what unlocks it.

Coordination Reducing Design

The lesson isn't Pony-specific: "act locally, gossip don't coordinate" is a systems design principle. Pony just applies it more consistently than most, all the way down to memory management. If that sounds interesting — here's where to find us.
