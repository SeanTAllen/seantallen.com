Coordination Pain

  • The actor model conflates behavior and data — they're bound together in each actor.
  • What happens when you want one operation over more than one set of data?
  • You either normalize upfront so the data lives in one actor — or you jump through coordination hoops.
  • For most use cases, this doesn't matter.
  • For things like an ACID database, it does.

Actors on ACID

Each table is an actor

Users
Orders
Inventory
  • Each table modeled as its own actor.
  • Access to the table is mediated by its actor — the actor is the only way in.
  • Why this shape? Horizontal scaling — each actor runs independently.
  • Single-actor DB is the alternative, but it defeats the point.

A simple query

sequenceDiagram participant C as Client participant U as Users actor C->>U: get user 42 U-->>C: user 42 data
  • Query is a message to the actor.
  • Result is a message back.
  • No problem — this is what actors are great at.

Another simple query

sequenceDiagram participant C as Client participant O as Orders actor C->>O: get orders for user 42 O-->>C: order list
  • Same pattern, different actor.
  • Still no problem.

A JOIN

sequenceDiagram participant C as Client participant U as Users actor participant O as Orders actor C->>U: get user 42 C->>O: get orders for user 42 U-->>C: user 42 data O-->>C: order list Note over C: combine into JOIN result
  • Now we need data from two actors.
  • Looks fine on the surface — but consistency is the question.
  • What if Users changes between the two responses?

You'd have to…

  • Pause incoming messages on each table.
  • Coordinate the access order.
  • Release when done.
  • Handle failures and livelocks.
  • You're building a distributed mutex out of messages.
  • Slower and more fragile than what databases do internally.
