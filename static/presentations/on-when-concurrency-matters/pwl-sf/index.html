  • All variables are “protected” by an actor — no globals.

  • Stay with what the actor model is good at and everything is wonderful.
  • But there are points of pain — places it won't help.

Coordination Pain

  • The actor model conflates behavior and data — they're bound together in each actor.
  • What happens when you want one operation over more than one set of data?
  • You either normalize upfront so the data lives in one actor — or you jump through coordination hoops.
  • For most use cases, this doesn't matter.
  • For things like an ACID database, it does.

A naive JOIN

Three entities: the Client circle (turquoise) on the left, the Users actor (terracotta) at the top right, and the Orders actor (sage) at the bottom right. The actors are drawn in a slimmed-down form — same iconography (circle = actor, rectangle = the table it owns), just smaller. The Client wants data that lives in two different actors — that's a JOIN.

A naive JOIN

The Client sends a single async message to the Users actor: a request for user 42. Once sent, the message is gone — this isn't a synchronous call held open waiting for a return value.

A naive JOIN

The first message is gone — it has been delivered. The Client now sends a second, independent async message, this time to the Orders actor, asking for the orders that belong to user 42. Two separate sends, in some order, neither blocking the other.

A naive JOIN

Time has passed. Orders responds first — before Users. Response order is not controlled by the sender; whichever actor finishes processing first sends back first. The Client now holds the Orders payload, but the Users response hasn't arrived.

A naive JOIN

More time passes. Users finally responds with the user record. The earlier Orders response is no longer on screen — it already happened. The Client is now holding both pieces, but they arrived as two separate events, at different times.

A naive JOIN

No arrows on screen. The Client (highlighted with a pulsing ring) is doing the JOIN itself — combining the user record with the orders list locally, in code. There was no JOIN message; the actors didn't do it. The "JOIN" is whatever the Client decides to do with the two pieces it received over time.

First scenario. Nobody else is touching the database while our JOIN runs. The simplest possible interleaving.

Our Client reads Users first. Yellow ring marks the moment. Other Writer's timeline is empty — nothing concurrent is happening.

Alice plus 2 orders. Matches the baseline. This is the answer the audience expected. Our isolation property held only because nothing else was happening.

Other Writer
adds order_C

Second scenario. While our JOIN is in flight, another client adds a new order to user 42's record. This isn't a fault — it's normal application traffic.

But the JOIN is not atomic

Our Client reads Users. At this point Users sees Alice. Order_C doesn't exist yet.

But the JOIN is not atomic

While our Client is between reads, Other Writer adds order_C to the Orders table. We have no idea this happened.

But the JOIN is not atomic

Our Client reads Orders — and Orders now has 3 rows because order_C landed. Our snapshot has Users from before the INSERT and Orders from after. That's an incoherent snapshot.

JOIN result:
Alice + 3 orders

Alice plus 3 orders. Different from Scenario A. The key thing to call out: this isn't a snapshot of any single moment in the database's life. It's a stitched-together view that mixes pre-INSERT Users with post-INSERT Orders.

Other Writer
removes order_B

Third scenario. Same setup, but this time the other client deletes one of the existing orders mid-JOIN.

But the JOIN is not atomic

Our Client reads Users first. order_B still exists at this point.

But the JOIN is not atomic

Other Writer removes order_B between our two reads. Again, our Client has no idea.

But the JOIN is not atomic

Our Client reads Orders. Now Orders only has 1 row because order_B is gone. Snapshot stitches pre-DELETE Users with post-DELETE Orders. Different mismatch, same shape of problem.

JOIN result:
Alice + 1 order

Alice plus 1 order. A third different answer from the same query.

Same JOIN.
Three different answers.

Same JOIN, three different interleavings, three different answers. The Client cannot predict which one it will get on any given run. This is not "wrong value" — it's "no isolation." And these are gentle examples, just an order count off. With financial transfers, inventory decrements, scheduling, the gap between "answer the JOIN gave" and "what was actually true at any single moment" can spin out of control fast. That's why we need locks.

Locks

Section divider. To make a JOIN atomic across multiple actors, we need to coordinate access — we need locks. Specifically, we need a distributed locking protocol because the data isn't in one place.

Two-phase locking

Sequence-diagram view. Three lifelines top to bottom: Client (turquoise) on the left, Users (terracotta) in the middle, Orders (sage) on the right. Time flows downward. Messages will appear as horizontal arrows between lifelines as the protocol unfolds.

Two-phase locking

Phase 1 begins: acquire locks. The Client sends a LOCK request to the Users actor for this transaction (txn=42). Until the lock is granted and acknowledged, nothing else can read or write the Users table for this transaction's duration.

Two-phase locking

The Client also sends a LOCK request to Orders. Both requests are outstanding. The lock acquisition order (Users before Orders, alphabetical) is fixed across all clients to prevent deadlock — if every client locked in the same order, no two could be waiting on each other.

Two-phase locking

Users grants the lock and replies. The Client now holds the Users lock but is still waiting on Orders. It can't proceed yet — both locks must be held before any read happens, or the JOIN won't be atomic.

Two-phase locking

Orders also grants. Both locks held. End of phase 1: four messages just to get permission to read. Now the Client can finally do the actual JOIN work.

Two-phase locking

Phase 2: actually read the data. The Client sends a READ to each actor. The locks held during phase 1 mean the data is stable — no one else can mutate either table while we're reading.

Two-phase locking

Both actors return their data. The Client now has consistent rows from both tables and can stitch them into the JOIN result. But the protocol isn't done — the locks are still held.

Two-phase locking — release

Phase 3: release the locks. Two RELEASE messages out, two RELEASED acknowledgments back. Until these complete, every other client trying to touch Users or Orders is blocked. Held locks are a global serialization point.

12 messages
for one JOIN.

Tally: 4 messages to acquire the locks, 4 to read the data, 4 to release the locks. Twelve messages of coordination noise around what was conceptually one query. And we haven't even talked about what happens if a lock holder crashes, or if the protocol picks the wrong order and deadlocks.

BOC

Section divider. Behaviour-Oriented Concurrency — or BOC for short, the abbreviation the audience will see for the rest of the talk. The deck title introduced the full term; this is where we start using it.

The actor model,
turned around.

The framing. Sets up the audience's mental model before the code lands. In the actor model behavior and state are bound together — coordination across them becomes a protocol. BOC pulls them apart: the data (cowns) and the work (behaviours) are separate, and coordination is built into how they get put back together. Same primitives, rotated.

What if the JOIN just looked like this?

when (users, orders) {
  let u = users.find(42)
  let o = orders.where(u.id)
  return JOIN(u, o)
}

The payoff. The JOIN we just spent 12 messages of protocol on, written as one block. The when keyword takes the cowns it needs, the body runs with exclusive access to all of them, and the result drops out. No LOCK / GRANTED / READ / DATA / RELEASE / RELEASED choreography. The whole protocol got absorbed into one keyword.

Cowns
own data

"Cown" = concurrent owner. A unit of state — like the data an actor owned, but pulled out from behind the actor. Each cown protects its contents; only one behaviour at a time can hold it. Think of them as the actor's table, no longer attached to a behavior.

Behaviours
acquire cowns

Behaviours are the units of concurrent work. A behaviour names the cowns it needs, the runtime grants them atomically (all at once or not at all), the behaviour runs with exclusive access to all of them, then releases. This is how work happens in BOC.

Atomic across cowns.

The new addition — the thing actors don't give you. A single behaviour holds multiple cowns at once, atomically, with no protocol. The missing piece that lets ACID semantics fall out.

Actor gifts.
Plus ACID.

The thesis, stated up front. We kept the actor gifts and added ACID. The refrain that follows backs up the "actor gifts" half — here are the gifts, one by one. Not "an ACID DB in Pony" — BOC is Verona, not Pony — but the means to build an ACID database without giving up what the actor model gave us.

What carries over?

Sets up the recap. Everything that follows is a callback to the actor goodness slides from earlier — the point is that BOC keeps all of it. The "still true" tag on each slide tells a cold reader this is a recap, not backward navigation.

Cowns
receive messages

Cowns receive messages — each cown has a mailbox, same as an actor. The affordance carries over: you still send a message to reach the data. What changed is what receipt means — we come back to that at the end.

Cowns
protect resources

Cowns protect their contents — files, sockets, data structures — same as actors did.

Cowns
rent their memory

A cown's memory is rented, not owned outright. A behaviour holds it for the length of its run, then gives it back. One tenant at a time.

No
globals

No
locks

No user-visible locks. The runtime acquires cowns for you; you never write a lock.

No
shared state

Behaviours allow
sequential reasoning

Inside a behaviour you reason sequentially — one step after another, with exclusive access to your cowns.

Behaviours can
scale horizontally

Behaviours over disjoint cowns run in parallel — same horizontal scaling, finer-grained sharing.

Behaviours are
deadlock-free

A behaviour acquires all its cowns at once, up front. No partial-hold-and-wait, so no deadlock.

Behaviours provide
data race freedom

A behaviour has exclusive access to its cowns while it runs. No two behaviours touch the same memory at the same time.

Actor Model:
messaging is
execution

The full thing: message reception is execution in the actor model. Receiving a message and running are the same event — the actor executes the moment the message arrives.

BOC:
messaging is
a ticket for execution

The full thing: message reception is a ticket for a cown to be used in an execution by a behaviour. The message doesn't run anything on its own — it claims the cown, and a behaviour later runs using that cown.

The behaviour
is the executor.

The title payoff. The thread moved out of the actor and into the behaviour. The behaviour is the logical thread now — it's what runs, holding its tickets until it can. The executor is the behaviour, not the actor. That's the "behaviour-oriented" in Behaviour-Oriented Concurrency. Defer the rest — how tickets get ordered, when a behaviour actually fires — that's a talk of its own.
